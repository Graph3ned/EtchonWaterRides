RELATIONSHIP SUMMARY:
====================

1. USERS → SESSIONS (One-to-Many)
   - Foreign Key: sessions.user_id → users.id
   - Cascade: SET NULL on user deletion

2. USERS → RIDES (One-to-Many)
   - String-based: rides_rental_dbs.user → users.name
   - No foreign key constraint

3. PRICES → RIDES (One-to-Many)
   - Implicit: rides_rental_dbs.rideType + classification → prices.ride_type + classification
   - No foreign key constraint

DESIGN PATTERN: LOOSE COUPLING
==============================

✅ STRICT FK RELATIONSHIPS:
   - sessions.user_id → users.id (for cleanup)

✅ LOOSE COUPLING RELATIONSHIPS:
   - rides_rental_dbs.user → users.name (for data preservation)
   - rides_rental_dbs.rideType + classification → prices.ride_type + classification (for flexibility)

BENEFITS:
- Data preservation (historical rides never lost)
- Flexibility (user names can change)
- Performance (no FK constraint checks)
- Simplicity (easier to maintain)
